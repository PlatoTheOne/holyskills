# 多语言翻译总方案（Lenny Data）

## 1. 目标
- 让站点在 `zh-CN`、`zh-HK`、`en` 三种语言下都具备一致体验：
- 界面文案：已支持三语。
- 元数据：标题、摘要、副标题、嘉宾、标签，需要分批翻译并上线。
- 正文内容：按批次翻译，支持逐步覆盖，不要求一次完成。

## 2. 当前规模（基于 2026-03-18 数据）
- 文档总量：638 篇
- 其中播客：289 篇
- 其中 Newsletter：349 篇
- 总词数：5,433,684 词
- 平均每篇：约 8,517 词

结论：正文翻译必须批次化推进，否则周期和质量不可控。

## 3. 批次策略（建议 24 批）
- 24 批每批约 22-30 篇，按总词数均衡分配。
- 先翻元数据（标题/摘要/标签）再翻正文。
- 每批执行顺序：
1. 翻译元数据（快）
2. 校对与术语统一
3. 上线该批元数据（用户可立即看到多语言标题/标签）
4. 翻译正文
5. 正文 QA 后上线

## 4. 信达雅 + 通俗化标准
- 信：事实与逻辑不失真，术语含义不偏移。
- 达：中文表达流畅，信息结构清晰。
- 雅：语体专业但不艰涩，避免直译腔。
- 通俗化原则：
1. 优先让读者看懂，再追求“字面对齐”。
2. 长句拆短句，术语首次出现可加简短解释。
3. 避免过度口语化，保持可引用的专业表达。

## 5. 术语与标签统一
- 建立 `tag_map`（标签词典），禁止同一标签多译并存。
- 建议先冻结高频标签译法（如 `design`, `leadership`, `strategy` 等）。

## 6. 已落地的工程能力
- 支持加载 `i18n/<locale>/meta.json` 覆盖标题、摘要、副标题、嘉宾。
- 支持标签翻译 `tag_map`。
- 支持读取 `i18n/<locale>/search-index.json` 做本地全文检索（按语言）。
- 支持正文按语言加载：
  - 本地私有模式：优先读取 `data/i18n/<locale>/docs/<filename>`
  - 私有 API 模式：`/content?filename=...&locale=...`
- 数据同步脚本已支持复制/构建 `i18n` 目录及本地化搜索索引。

## 7. 操作命令（批次化）
1. 初始化某语言批次任务（例：简体中文，24 批）：

```powershell
node .\scripts\init-translation-batches.mjs --locale zh-CN --batches 24
```

2. 翻译人员逐批填写：
- `translation/zh-CN/batches/batch-01.json`
- 把 `translated` 字段补齐，`status` 改为 `done`
- 标签译名填入 `translation/zh-CN/tag-map.json`

3. 编译为站点可读的翻译元数据：

```powershell
node .\scripts\compile-translation-meta.mjs --locale zh-CN
```

生成：
- `lennys-newsletterpodcastdata-all/i18n/zh-CN/meta.json`

4. 同步到前端数据目录：

```powershell
node .\scripts\sync-lenny-data.mjs --mode public
```

## 8. 建议的执行节奏
- 第 1-2 周：先完成 24 批元数据翻译（标题+摘要+标签），快速改善全站体验。
- 第 3 周起：正文按优先级翻译（先高流量、高价值、常访问主题）。
- 每周固定发布 2-3 批，持续可见进度。

## 9. 质量门槛（发布前）
- 元数据抽检通过率 >= 98%
- 术语一致性 >= 99%
- 正文关键事实错误 = 0
- 标题可读性评分（人工）>= 4/5
